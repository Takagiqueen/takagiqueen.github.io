## 最开始，我只是觉得一句“生日快乐”有些单薄

但普通的图片似乎少了一点过程，聊天框里的一句“生日快乐”又很容易被新的消息推走。我希望对方收到的不只是几行文字，而是一段有意思的体验。

于是，我开始设想：

先看到一封被火漆封存的信，轻触封印后，贺卡展开；光芒掠过纸面，隐藏的祝福一点点浮现；读完之后，画面再进入最后的生日祝福。

我把它叫作——**封存的花信**。

第一版构思里有深蓝色贺卡、金色火漆、蓝粉蝴蝶、羊皮纸、荧光笔和粒子消散。

这些元素单独看都很漂亮，放在一起却显得有些拥挤。深蓝与金色偏庄重，蝴蝶偏童话，荧光笔又很现代，整个画面缺少统一的情绪。

后来我逐渐意识到，网页动画不是越多越好。

我最终把视觉调整为低饱和的桑葚红、暖白信纸和旧香槟金，减少粒子数量，放弃让整个贺卡“爆炸式消散”。火漆只是轻轻裂开，纸页慢慢展开，装饰元素只在必要时出现。

## 技术使用

项目使用 Vite、TypeScript、GSAP、Canvas 2D 和 CSS 3D Transform 实现。

CSS 3D 负责贺卡翻开的空间感，GSAP 将火漆裂开、卡页展开、信纸出现、显影完成和终章串联在同一条动画时间轴中。

我认为比较特别的是“寻光显影”。

祝福文字实际一直存在于页面的 DOM 中，Canvas 只是在文字上方覆盖了一层与信纸颜色相同的遮罩。用户移动鼠标或手指时，程序擦除经过区域的遮罩，让文字逐渐显现。

这样做既保留了互动效果，也避免让 Canvas 成为文字的唯一载体。即使 Canvas 不可用，祝福仍然能够被读取。

在移动端，我还需要处理高清屏缩放、快速滑动留下的轨迹断点、页面滚动冲突，以及不同屏幕尺寸下的显影进度计算。

为了避免用户必须把整张纸反复涂满，我没有检测所谓的“最后一个字”，而是记录有效文字区域的覆盖程度。达到一定比例后，剩余内容会自动补全。

## 从“我的贺卡”到“任何人的贺卡”

项目完成以后，我最初只是准备把网页链接发给别人。

但在整理代码时，我发现姓名、日期、祝福和颜色已经可以集中配置。既然如此，它就不必只是一张属于我的贺卡，也可以成为其他人制作祝福的起点。

于是我继续做了几件事：

- 将所有可修改内容集中到一个配置文件；
- 使用 GitHub Actions 自动构建并部署；
- 将仓库设置为 Template Repository；
- 编写不需要本地开发环境的使用教程。

现在，使用者只需要点击 `Use this template`，修改姓名、日期和祝福文案，再开启 GitHub Pages，就能得到属于自己的网页贺卡。

## 意外的收获

发布分享帖时，我没有预料到它会得到太多关注。

截至写下这篇文章时，帖子已经收到了 300余个收藏与点赞。

这个数字当然不能证明项目已经有多成熟，但它让我第一次比较直接地感受到：一个很小的创意，只要被完整地做出来，也能对别人产生特别的价值。

有人或许想把它送给朋友，有人想修改后送给喜欢的人，也有人只是觉得这种表达祝福的方式很有意思。

对我来说，最有意义的并不是数字本身，而是原本只属于一个人的小想法，开始被更多人带去表达他们自己的心意。
